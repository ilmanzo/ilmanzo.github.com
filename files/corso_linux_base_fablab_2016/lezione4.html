7/47

stati di un processo

  • 'D' = uninterruptible sleep
  • 'R' = running
  • 'S' = sleeping
  • 'T' = traced or stopped
  • 'Z' = zombie

'R' is the easiest; the process is ready to run, and will run whenever its turn to use the CPU comes.

'S' and 'D' are two sleep states, where the process is waiting for something to happen. The difference is that 'S' can be interrupted by a signal, while 'D' cannot (it is usually seen when the process is waiting for the disk).

'T' is a state where the process is stopped, usually via SIGSTOP or SIGTSTP. It can also be stopped by a debugger (ptrace). When you see that state, it usually is because you used Ctrl-Z to put a command on the background.

'Z' is a state where the process is dead (it has finished its execution), and the only thing left is the structure describing it on the kernel. It is waiting for its parent process to retrieve its exit code, and not much more. After its parent process is finished with it, it will disappear.

permessi ESTESI

ci sono altri 3 bit per i permessi speciali: STICKY, SUID, SGID

# chmod +t /usr/local/tmp

lo sticky bit viene comunemente usato per le directory destinate a contenere file temporanei di pi utenti: se impostato su una directory esso indica che i file e subdirectory in essa contenuti possono essere cancellati o rinominati solo dal proprietario, o dal proprietario della directory che lo contiene o dall'utente root, anche se si dispongono di tutti gli altri permessi di scrittura necessari.

25/47

permessi ESTESI

SUID e SGID servono per modificare il contesto di esecuzione di un file eseguibile

chmod u+s /usr/local/bin/mioprog
chmod u+g /usr/local/bin/altroprog

l'eseguibile "SUIDato" viene lanciato nel contesto dell'utente proprietario (anziche' quello di chi l'ha eseguito)

ad esempio, in linux il comando /bin/ping e' di proprieta' root ed ha il bit suid attivo: cio' significa che viene sempre eseguito come se fosse lanciato dall'utente root

SGID si usa ad esempio per creare delle directory "condivise" da un certo gruppo di utenti eterogenei

  • creo gruppo "classe" vuoto
  • assegno agli utenti il gruppo secondario "classe"
  • creo la directory "shared" con SGID bit, user owner "root" e group owner "classe"
  • cosa succede quando un utente (andrea) scrive un file nella directory ?
  • cosa succederebbe se la directory non avesse il bit SGID ?

oltre i permessi ESTESI: ACL

oltre ai normali permessi, su alcuni filesystem che le supportano (se vengono montati con l'opzione apposita) possiamo usare le POSIX Access Control List (ACL) per definire con la massima granularita' gli accessi

  • setfacl

    setfacl -m "u:andrea:rwx" miofile

  • getfacl

    getfacl miofile

quando un file ha ACL associate, nell'output di ls -l appare un '+'

dev e udev

ogni device e' identificata nel sistema con una coppia di numeri: il major e minor number. Vediamone alcuni ...

ls -l /dev

nelle versioni recenti di Linux c'e' un demone (udevd) che si occupa di creare al volo i file necessari del tipo giusto con i permessi giusti quando una periferica viene aggiunta o rimossa dal sistema

come ogni altra parte del sistema operativo, anche udevd e' configurabile , in particolare sono importanti le rules , vediamone alcune

ls -l /etc/udev/*

ad esempio si possono modificare queste regole se vogliamo che una certa scheda di rete con un certo mac-address si chiami in un certo modo (e sempre quello) a ogni avvio del sistema

# udevadm info /dev/sda

http://www.reactivated.net/writing_udev_rules.html

quale filesystem scegliere ?

esistono diversi tipi di filesystem, ognuno ha pregi e difetti, e risulta piu' adatto a cose specifiche; qualche esempio:

  • VFAT, NTFS (mondo Windows)
  • HFS (sui sistemi Apple)
  • ext3,ext4 : il fs tradizionale linux
  • xfs, btrfs : il nuovo che avanza
  • zfs: per sistemi enterprise (Oracle)
  • unionfs, jffs2: per esigenze specifiche
  • nfs, smbfs: accesso a dati in rete

eccetera https://en.wikipedia.org/wiki/Comparison_of_file_systems

mount, umount

con questi comandi possiamo aggiungere o togliere a piacimento (cioe' "innestare" e "potare" ) interi rami dall'albero durante il funzionamento del sistema

nelle distribuzioni moderne cio' viene fatto automaticamente (ad esempio quando inserisco una pennetta USB) grazie a un servizio (demone)

come ogni comando, mount ha molte opzioni che modificano il suo comportamento qualche esempio:

  • user
  • noauto
  • noatime
  • sync
  • noexec
  • ro
